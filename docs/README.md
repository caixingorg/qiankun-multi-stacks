# 文档总入口

## 一、先看这个

如果你只想知道“现在应该看哪份文档”，不要直接在 `docs/` 里按文件名猜。

请按下面 3 类理解：

1. **永久基线文档**：长期有效，回答“系统现在是什么样”
2. **过程文档**：某次评审、某轮优化、某个需求落地过程的记录
3. **历史归档**：保留追溯价值，但不再作为当前实施依据

## 五、推荐阅读顺序

### 场景 A：第一次接手仓库

1. 先看 [架构文档索引](./architecture/README.md)
2. 再看 [当前架构图与挂载流程](./architecture/2026-04-04-当前架构图与挂载流程.md)
3. 再看 [主应用基座与接入基线说明](./architecture/主应用基座与接入基线说明.md)
4. 再看 [main 应用文件夹说明](./architecture/main-应用文件夹说明.md)
5. 最后按需要补看契约类文档

### 场景 B：想知道最近改了什么

1. 先看 [当前项目架构评估报告](./reviews/2026-04-04-当前项目架构评估报告.md)
2. 再看 [微前端架构优化计划二次评估](./reviews/2026-04-05-微前端架构优化计划二次评估.md)
3. 再看 [main 运行逻辑瘦身评估](./reviews/2026-04-05-main-运行逻辑瘦身评估.md)

### 场景 C：只想知道哪些文档能长期信

只看：

1. `docs/architecture/`
2. 这个文件里列出的“永久基线文档”

不要默认把 `plans/`、`reviews/`、`tasks/` 当成长期规范。
