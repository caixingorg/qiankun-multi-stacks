# 主应用基座与接入基线说明

## 一、目标

这份文档只回答一个问题：

**未来子应用团队应该基于 `main` 主应用的哪些能力接入，而不应该自己发明哪些能力。**

接入操作清单请优先看：

- [子应用接入短清单](./子应用接入短清单.md)

## 二、主应用真正拥有的能力

`main` 负责的平台能力包括：

1. 子应用注册与入口解析
2. 运行时配置与环境配置读取
3. 主路由激活规则
4. 全局权限上下文
5. 全局导航能力
6. 主子应用最小通信通道
7. 错误上报与回滚决策
8. 共享 request 能力与 runtime vendor
9. 主应用单一上下文状态源

## 三、子应用允许依赖的能力

子应用接入时可以依赖：

1. 主应用通过 qiankun `props` 下发的：
   - `actions`
   - `bus`
   - `sharedKernel`
   - `userContext`
   - `envContext`
   - `permissionContext`
   - `navigation`
   - `dependencyPolicy`
   - `contractVersion`
2. 子应用自己仓内的 `src/bridge/` 文件：
   - `micro-app.js`
   - `standalone.js`
   - `host-api.js`
   - `get-host-context.js`

明确禁止：

1. 子应用直接 import `main/src/*`
2. 子应用把 `main/src/runtime/*` 当作可复用 SDK
3. 子应用把 `main/src/shared/*` 当作跨仓稳定入口

补充约束：

1. `userContext / envContext / permissionContext / contractVersion` 应来自主应用同一套上下文实例
2. 非本地环境如果缺失 manifest 或 auth 注入，主应用应直接暴露 host 级失败，而不是自动回退到开发态

## 四、子应用不允许控制的内容

子应用不得自己决定：

1. 一级菜单
2. 主应用激活路由
3. 发布通道切换
4. rollback 决策
5. 全局权限模型
6. 主应用运行时配置来源

## 五、主应用 public 资源基线

`main/public` 当前按 3 层组织：

```text
main/public/
├── runtime/
│   ├── runtime-config.js
│   └── runtime-vendor.global.js
└── config/
    ├── env/
    └── apps/
```

含义：

1. `runtime/`：浏览器启动即加载的 host runtime 资源
2. `config/env/`：环境级 host 配置
3. `config/apps/`：子应用 manifest

## 六、新子应用接入最小步骤

### 1. 声明 app registry 信息

在 host manifest 或 registry 中补：

1. `key`
2. `name`
3. `activeRule`
4. `entries`
5. `rollbackMode`

### 2. 接入主应用契约

子应用 bridge 应在本地 `host-api.js` 中承接：

1. `getHostSharedKernel`
2. `getHostNavigation`
3. `getDependencyPolicy`
4. `assertMainContract`

### 3. 使用 host 生命周期辅助

子应用应在自己仓内统一走本地 lifecycle helper：

1. `createWebpackSubapp`
2. `createViteSubapp`
3. `createCleanup`
4. `createBridge`

### 4. 仅消费 host 提供的导航能力

子应用内部二级路由可以自己管理，但跨应用跳转应统一走 `navigation.navigate()`。

## 七、判断接入是否合格

一个子应用如果满足以下条件，就算接入合格：

1. mount / unmount 明确
2. 副作用可以被 cleanup 清理
3. 不接管 host 一级导航
4. 不自行决定 rollback 与发布通道
5. 通过 host contract 接入上下文、请求、导航与错误上报
