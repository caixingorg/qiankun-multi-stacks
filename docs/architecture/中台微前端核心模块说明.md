# 中台微前端核心模块说明

## 一、三层功能划分

本次规划按三层理解：

1. 整体性核心功能
2. 主应用核心功能
3. 子应用核心功能

## 二、整体性核心功能

这一层优先级最高。

包括：

1. 统一工程化入口
2. 统一依赖管理
3. 主应用与子应用联调机制
4. 统一构建与发布入口
5. 统一校验与 smoke 验证
6. 统一基础 UI 组件使用边界
7. 统一目录规范
8. 登录认证与权限模型
9. 配置中心与环境配置
10. 请求层规范
11. 错误监控与日志埋点
12. 菜单、路由、权限联动关系

当前这些能力的说明已经分别落在：

1. `docs/architecture/登录认证与权限模型说明.md`
2. `docs/architecture/请求层规范说明.md`
3. `docs/architecture/错误监控与日志方案说明.md`
4. `docs/architecture/菜单路由权限联动说明.md`

## 三、主应用核心功能

主应用统一命名为：

```text
main/
```

主应用核心功能包括：

1. 应用壳
2. 子应用注册与加载
3. 路由分发
4. 用户 / 环境 / 权限上下文
5. 菜单与导航
6. 最小通信能力
7. 基础运行治理

### 主应用完整目录结构

```text
main/
├── src/
│   ├── main.js
│   ├── app-registry.mjs
│   ├── layout/
│   ├── components/
│   ├── pages/
│   ├── routes/
│   ├── context/
│   ├── navigation/
│   ├── communication/
│   ├── runtime/
│   ├── contracts/
│   ├── shared/
│   ├── services/
│   ├── constants/
│   ├── utils/
│   └── styles/
├── public/
├── index.html
├── package.json
└── vite.config.js
```

### 主应用各模块说明

- `layout/`：主壳布局
- `components/`：主壳基础组件，允许使用 `shadcn/ui`
- `pages/`：主应用自身页面
- `routes/`：主应用路由与子应用激活规则
- `context/`：用户、环境、权限上下文
- `navigation/`：菜单与导航跳转
- `communication/`：最小通信能力
- `runtime/`：加载、错误、卸载等运行保障
- `contracts/`：主应用自己的契约定义，不再单独抽 `packages`

## 四、子应用核心功能

所有子应用统一建议放在：

```text
subapps/
```

每个子应用核心功能包括：

1. 子应用入口
2. 子应用内部页面
3. 业务模块
4. 子应用状态
5. 服务层
6. 基础组件
7. 与主应用对接的桥接层

### 子应用完整目录结构

```text
subapps/app-name/
├── src/
│   ├── main.js
│   ├── app.js
│   ├── routes/
│   ├── pages/
│   ├── modules/
│   ├── components/
│   ├── bridge/
│   ├── stores/
│   ├── services/
│   ├── constants/
│   ├── utils/
│   └── styles/
├── public/
├── index.html
├── package.json
└── 构建配置
```

### 子应用模块说明

- `pages/`：页面级视图
- `modules/`：业务模块
- `components/`：基础页面组件
- `bridge/`：与主应用对接
- `stores/`：子应用自身状态
- `services/`：接口与服务层

## 五、共享层说明

共享能力已经收回主应用内部：

```text
main/src/shared/
├── constants/
├── host-adapters/
├── ui/
└── utils/
```

这里明确去掉一个旧做法：

1. 不再保留 `packages/` 这种独立 SDK 抽象层
2. `runtime-vendor` 已经回并到 `main/src/runtime/runtime-vendor/` 与 `main/src/shared/host-adapters/`

## 六、主子应用契约

企业级微前端模板必须补齐这些契约：

1. 菜单契约
2. 路由契约
3. 权限契约
4. 上下文契约
5. 通信契约
6. 错误契约

## 七、发布与部署模型

企业级模板必须补齐这些内容：

1. 环境划分
2. 入口来源
3. 构建产物
4. 回滚模型
5. 版本兼容要求

## 八、UI 使用边界

当前确认：

- UI 可以使用 `shadcn/ui`

但边界明确如下：

1. 主应用和 React 子应用可优先采用 `shadcn/ui`
2. Vue 子应用不强制依赖 `shadcn/ui`
3. 先解决“丑”和“不统一”
4. 不在这一轮扩成完整设计系统
